Real-Browser Accessibility Auditing

Your AI agent checks contrast ratios, tab order, focus indicators, ARIA labels, and semantic HTML against WCAG 2.1 AA — in a real browser with real rendering.

How it works

Four phases. One comprehensive audit.

1

Reviews your codebase first

Before opening the browser, it scans for ARIA usage, semantic HTML, image alt text, form labels, and heading hierarchy in your source code.

2

Checks visual accessibility

Opens the page in a real browser and verifies color contrast ratios (4.5:1 for normal text, 3:1 for large), font sizes, and touch target dimensions.

3

Tests keyboard navigation

Tabs through the entire page to verify tab order, focus indicators, focus traps in modals, skip links, and keyboard operability of interactive elements.

4

Validates ARIA and semantics

Verifies landmarks, form label associations, dynamic content announcements with aria-live, accessible names for images and icons, and table headers.

5

Screenshots every issue

Each finding is captured with a screenshot showing the exact element, its location, the WCAG criterion violated, and the impact on users.

6

Delivers a prioritized report

Issues are categorized as Critical, Serious, Moderate, or Minor. Each includes a specific fix, file location, and complexity estimate so you can triage fast.

Why this is different

Not just a static linter.

Real browser, real rendering

Static analysis tools miss computed contrast, dynamic focus states, and JavaScript-driven ARIA. This audits what users actually experience in a real Chrome instance.

Code-aware remediation

Each issue maps back to your source files with a specific fix and complexity estimate. Not just "fix contrast" but which file, which element, and what value.

Works with your existing tools

Claude Code, Codex, Cursor, and other MCP clients can run it. No custom a11y framework required. Integrates into your development workflow as-is.
